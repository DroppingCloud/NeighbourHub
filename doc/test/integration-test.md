# 联调测试记录

## 1. 测试目的

用于验证系统在真实运行环境下的端到端业务闭环，包括登录注册、事项办理、材料上传与预审、工作人员审核、服务预约、家属代办和通知展示。

自动化单元测试主要隔离数据库和外部服务；本文件记录需要在完整环境中人工验证的核心流程，作为课程提交中的核心功能测试补充。

## 2. 测试环境

- 前端：Vue 3 + Vite，默认地址 `http://localhost:5173`
- 后端：Spring Boot，默认地址 `http://localhost:8080`
- 数据库：MySQL 8，数据库名 `community_service`
- 缓存：Redis 7
- 文件存储：本地材料上传目录
- OCR/AI：DashScope OCR，可通过环境变量控制是否启用

## 3. 测试前准备

1. 启动 MySQL 与 Redis。
2. 使用 `doc/database/init.sql` 初始化数据库，并确保导入时使用 `utf8mb4` 字符集。
3. 启动后端服务。
4. 启动前端服务。
5. 使用管理员账号创建或确认存在事项工作人员、服务预约工作人员、居民用户和家属用户。

## 4. 核心联调用例

| 编号 | 测试场景 | 操作步骤 | 预期结果 |
| --- | --- | --- | --- |
| IT-01 | 登录注册 | 注册居民用户并登录；使用管理员和工作人员账号登录 | 登录成功后进入对应角色首页，菜单和权限正确 |
| IT-02 | 居民事项办理 | 居民选择事项，填写申请信息，上传必需材料，提交申请 | 申请进入“我的申请”，后台生成工单并分配给事项工作人员 |
| IT-03 | 材料预审 | 上传格式正确材料、格式错误材料、明显不相关材料 | 正确材料通过；错误格式或不相关内容提示预审不通过 |
| IT-04 | 工单审核 | 事项工作人员查看工单详情、材料列表和完整性结果，审核通过或退回补件 | 状态同步更新到申请；居民端收到对应通知 |
| IT-05 | 补件重新提交 | 居民对待补件申请补充材料并重新提交 | 申请状态回到待审核，工作人员可继续处理 |
| IT-06 | 服务预约 | 居民提交助餐、陪诊或上门服务预约 | 预约保持待接取状态，服务预约工作人员可在工作台接单 |
| IT-07 | 预约接单权限 | 事项工作人员尝试接取服务预约；服务预约工作人员接取预约 | 事项工作人员被拒绝；服务预约工作人员可接取对应类型预约 |
| IT-08 | 家属代办 | 家属提交绑定申请，居民确认后，家属代办事项或预约 | 家属只能访问授权对象数据，申请/预约显示代办标识 |
| IT-09 | 管理员管理 | 管理员查看居民、家属、工作人员列表，新增工作人员 | 用户分类统计正确；新增工作人员可用初始密码登录 |
| IT-10 | 消息通知 | 完成申请提交、审核、预约接单等动作 | 对应居民、家属、工作人员或管理员收到通知 |

## 5. 已知风险

- OCR/AI 结果受外部服务、图片质量和网络状态影响，演示前需要提前准备稳定测试材料。
- 文件预览依赖浏览器对 PDF、图片和 Office 文档的支持，不同浏览器表现可能略有差异。
- 如果数据库不是用 `utf8mb4` 客户端导入，事项或材料中文数据可能出现乱码，需要重新初始化数据库。

## 6. 测试结论

自动化测试已覆盖核心后端逻辑；联调测试用于补充验证真实数据库、文件上传、OCR/AI、前端页面交互和多角色协作流程。演示前应至少完整跑通 IT-01 到 IT-08。
